Project Goal: A real-time "Meetup" platform where a host can broadcast video/audio to an audience.

Core Tech Stack:
Framework: Next.js 14+ (App Router, TypeScript).
Real-time Media: WebRTC via LiveKit (SFU architecture).
Styling: Tailwind CSS + Shadcn UI.
Database/Auth: Prisma (Postgres) and Clerk for user management.

Key Architecture: > Unlike a standard 1:1 video call, this app uses an SFU (Selective Forwarding Unit). The Host publishes tracks to the SFU, and the SFU distributes those tracks to many Viewers. This ensures the Host's upload bandwidth doesn't explode as the audience grows.

Role-Based Logic: > * Host: Full media permissions (Camera/Mic/Screen Share).

Viewer: Receive-only by default; can "Raise Hand" to request speech permissions.

------------------------------------------------------------

Phase 1: The Foundation (CRUD & Auth)
Goal: Build a dashboard where users can create and see "Meetup Rooms."

Learning: Next.js App Router, Prisma schemas, and Clerk integration.

Task: Create a /dashboard to list meetups and a /meetup/[id] page that checks if the user is the creator (Host) or a guest (Viewer).

Suggestion: Before building UI, lock the data model. Prisma: User (id, clerkId), MeetupRoom (id, title, hostId, createdAt, optional livekitRoomName). Role = host if MeetupRoom.hostId === current user. Run first migration so Phase 2 doesn’t require rework.

Phase 2: The "Signal" (WebRTC Handshake)
Goal: Securely connect a user to a video room.

Learning: JWTs (JSON Web Tokens) for media servers and how "Signaling" works.

Task: Create an API route /api/livekit/token that generates a token with canPublish permissions based on the user's role in the database.

Suggestion: Use room name e.g. meetup-{meetupId}. Token API: accept roomName/meetupId + participantIdentity (Clerk userId), look up meetup in DB, set canPublish only if user is host—never trust client for "I am host." Validate meetup exists and user is allowed to join.

Phase 3: The Media Stream (The "Aha!" Moment)
Goal: Get video on the screen.

Learning: MediaDevices API (camera access) and WebRTC "Tracks."

Task: Integrate @livekit/components-react. Use the <LiveKitRoom> and <VideoConference> components to get a working stream between two browser tabs.

Suggestion: Test with two different users/tokens: Tab 1 = host (canPublish), Tab 2 = viewer (subscribe-only). Confirm viewer sees host and cannot publish. That validates Phase 2 token logic.

Phase 4: Custom Interaction (The "Meetup" Polish)
Goal: Move beyond basic video chat.

Learning: WebRTC Data Channels (sending low-latency messages).

Task: Build a "Raise Hand" feature. A Viewer sends a data packet; the Host receives it and can "Accept," which triggers a permission update to let the Viewer turn on their mic.
---
Suggestion: Use LiveKit Server API to grant/revoke canPublish when Host accepts. Decide where state lives: (A) LiveKit + data channel only (simplest), or (B) DB table (e.g. RaiseHandRequest) if you want history/analytics. Start with (A) and add DB later if needed.
---
Phase 5: Production & Scaling
Goal: Deploy to the real world.

Learning: Latency optimization and STUN/TURN servers.

Task: Connect your app to LiveKit Cloud (managed SFU) and deploy the frontend to Vercel.

Suggestion: Set LIVEKIT_URL, LIVEKIT_API_KEY, LIVEKIT_API_SECRET (and DATABASE_URL, CLERK_*) in Vercel env. Keep token generation in a server-side API route only—never expose LiveKit API secret to the client.
